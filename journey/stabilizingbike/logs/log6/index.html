The simulation

I programmed this simulation in Rust (no surprise). I used kiss3D which is this really wonderful 2D/3D graphics library for quick and dirty visualizations. It defines a lot of graphics primitives so you don’t have to waste time doing it from scratch.

I ended up using quaternions for the first time! I don’t understand the full algebra and intuition behind them. But I found this article helpful to work through since its explanation is almost entirely self contained.

The most important thing I needed to know was how to rotate about an arbitrary axis. Suppose you have an axis of rotation represented by the unit vector $(a_x, a_y, a_z)$. The quaternion representing a rotation of $\theta$ radians about that axis is represented by the collection $(x, y, z, w)$

$$ \begin{align*} w&= \cos(θ / 2) \\ x&= a_{x} \sin(θ / 2) \\ y&= a_{y} \sin(θ / 2) \\ z&= a_{z} \sin(θ / 2) \end{align*} $$

Bike kinematics

For a rotating rigid body, like a bicycle, it has an instantaneous center of rotation (abbreviated by ICC). Another way of putting it, this is the point whose instantaneous velocity is zero. The location of the ICC relative to the bicycle is determined by the turn radius. If the bike is heading in a straight line, the radius of rotation approaches $\infty$.

The bike’s rotation, assuming no slip, is defined by the values in the table. Note that $\delta$ completely defines the radius of the turn with a fixed bike length.

Parameter Definition
$\delta$ The steering angle
$R$ The radius of the rotation about the ICC
$L$ The length of the bike
$V$ The forward velocity of the bike
$\omega$ The angular velocity of the bike about the ICC

This rotation about the ICC has two different interpretations. If viewed from the coordinate frame centered at the ICC, the bike’s motion is a pure rotation. If the rotation is viewed from the bike’s body frame, it appears as combination of an instantaneous forward translation and rotation about the bicycle frame.

This is essentially all my simulation is doing right now. For a small timestep, I rotate the bicycle frame and shift it forward by a small step according to the angular/linear velocity.

Here is the same explanation for rigid body transformations but in more mathematical terms. I apologize for all the vector symbols. Normally they’d be removed but it might be confusing for someone unfamiliar with the notation. Here is a notation table to reference while reading (don’t worry if you don’t understand what it means yet).

Variable Definition
$H_1 ^0$ Homogeneous/matrix transformation for rotation/translation to go from frame 1 to frame 0
$R_1 ^0$ A rotation matrix that performs a rotation to orient frame 1 onto frame 0.
$\vec{p}$ An abstract point in space not viewed from any particular frame.
$\vec{p}^i$ The same point in space but viewed from frame $i$
$\vec{d}_1 ^0$ Vector from the origin of frame 0 to the origin of frame 1
$\vec{v}_1 ^0$ Linear velocity of the origin of frame 1 expressed in frame 0
$\vec{\omega}(t)$ The angular velocity of the bike frame with respect to the fixed ICC frame

Let frame 0 be centered at the ICC and frame 1 centered at the bike’s body. How a point $\vec{p}^1$ in the bike frame (1) appears in the ICC frame (0) is encoded by the transformation $H_1 ^0$. $H_1 ^0$ is the combined transformation from frame 1 to frame 0. This is called a homogeneous transformation. It combines both rotations and translations into a single matrix transform.

$$ \vec{p}^0 = H_1 ^0 \vec{p}^1 = R_1 ^0 \vec{p}^1 + \vec{d}_1 ^0 $$

Subsequently, $R_{1}^{0}$ is the relative rotation from frame $O_{1}x_{1}y_{1}z_{1}$ to frame $O_{0}x_{0}y_{0}z_{0}$. $R_1 ^0$ performs a change of basis on the vector.

And $\vec{d_1^0}$ is the translation vector from the origin of frame $O_{0}x_{0}y_{0}z_{0}$ to the origin of $O_1 x_1 y_1 z_1$. $\vec{d}_1 ^0$ serves to shift the properly oriented vector so it points to the same point in space, but seen from a different perspective.

Lets take the time derivative of $\vec{p}^0=R_1 ^0 \vec{p}^1 + \vec{d}_1 ^0$. Use the product rule for the first term.

$$ \begin{align*} \frac{d}{dt} p^0 &= \dot{R}_1 ^0 \vec{p}^1 + R_1 ^0 \dot{p}^1 + \dot{d}_1 ^0 \\ &=S(\vec{\omega}(t)) R_1 ^0 \vec{p}^1 + R_1 ^0 \cancel{\dot{p}^1} + \vec{v}_1 ^0 \end{align*} $$ $\dot{p}^1$ cancels because it is a fixed point relative to the bike frame. $\vec{v}_1 ^0$ is the linear velocity of the frame.

If you take my word for it, the time derivative of a rotation matrix $R(t)$ about an axis by $\theta$ degrees is the same as multiplying it with a skew-symmetric matrix $S(\vec{\omega}(t))$ where $\omega(t)$ is the angular velocity of the rotating bike frame (same as axis of rotation).1

$$ \dot{R}(t) = S(\vec{\omega}(t)) R(t) $$

This is not critical to understand if you want the gist. It so happens that skew symmetric matrices have the property that multiplication with the vector is the same as a cross product (in three dimensions).

$$ S(\vec{\omega}(t)) \vec{p}^1 = \vec{\omega}(t) \times \vec{p}^1 $$

The final result is:

$$ \begin{align*} \frac{d}{dt}\vec{p}^0 &= \vec{\omega}(t) \times \vec{p}^0 + \vec{v}_1 ^0 \end{align*} $$

This indicates that the linear velocity of the point seen from the ICC frame depends on the angular velocity of the rotating bike frame plus the linear velocity the two frames have relative to each other.

The Code

There are a few essential parts of my code. There’s the main loop which initializes kiss3d and the scene. It continuously calls a step function to update the state of the simulation:

#[kiss3d::main]
async fn main() {
    let mut window = Window::new("Kiss3d: custom_mesh_shared").await;
    let mut camera = FirstPersonCamera3d::new(
        Vec3::new(0.0, 1.0, 5.0), // Eye position
        Vec3::ZERO,               // Looking at origin
    );

    // ROS REP-103 says forward is x, left is y, up is z
    camera.set_up_axis_dir(Vec3::new(0.0, 0.0, 1.0));

    let mut scene = SceneNode3d::empty();
    scene
        .add_light(Light::point(100.0))
        .set_position(Vec3::new(0.0, 10.0, 10.0));

    let circle = Rc::new(RefCell::new(create_circle_mesh(1.0, 150)));

    MeshManager3d::get_global_manager(|mm| {
        mm.add(circle.clone(), "circle_mesh");
    });

    let mut sim = BikeSim::new(&mut scene);
    println!("Init sim!");

    // Measure time since last loop and use that to
    let mut del_t = Duration::from_secs(0);
    loop {
        let t0 = Timestamp::now();
        sim.step(&mut window, &del_t);

        window.render_3d(&mut scene, &mut camera).await;
        let t1 = Timestamp::now();

        del_t = t1.duration_since(t0).unwrap();
        println!("del t={}", del_t.as_secs_f64());
    }
}

I make the distinction between two different types of information. The “state vector” of the bike which changes over time based on the kinematics and dynamics. And then fixed parameters of the bike, like its dimensions. The self.state.step() updates the state vector of the bike using the rotation about an instantaneous center equations.

The coordinate frames are then defined by the values within the state vector. Instead of directly transforming the coordinate frames when updating the simulation, I update the main angles of the bike (steering, roll, and yaw) and the BikeSim::bike_frames() function uses these angles to define the coordinate frames in 3D space.

pub fn step(&mut self, window: &mut Window, del_t: &Duration) {
    //println!("{:?}", self.state);
    // Do math to update bike state
    self.state
        .step(del_t, &mut self.frame_registry, &self.params);

    // Increment time
    self.state.t = Timestamp::now();

    // Update all the frames now that bike state has changed
    for f in BikeSim::bike_frames(&self.params, &self.state) {
        self.frame_registry.add_transform(f);
    }

    // Draw the bike onscreen using kiss3D
    self.viz
        .draw_bike(window, &mut self.frame_registry, &self.state);
}

To see what I mean, here is an example coordinate frame definition using the transforms crate.

fn bike_frames(params: &BikeParams, state: &BikeState) -> [Transform<Timestamp>; 9] {
    let bike_roll = Quaternion {
        // rotation along x axis
        w: f64::cos(state.bike_roll / 2f64),
        x: f64::sin(state.bike_roll / 2f64),
        y: 0.0,
        z: 0.0,
    };
    let bike_yaw = Quaternion {
        // rotation along z axis
        w: f64::cos(state.bike_yaw / 2f64),
        x: 0.0,
        y: 0.0,
        z: f64::sin(state.bike_yaw / 2f64),
    };

    let bike_frame = Transform {
        translation: Vector3::new(state.x, state.y, 0.0),
        rotation: (bike_roll * bike_yaw)
            .normalize()
            .expect("Failed to normalize bike frame rotations"),
        timestamp: state.t,
        parent: "world".into(),
        child: "bike_frame".into(),
    };

    let bike_com = Transform {
        translation: Vector3::new(2.0, 0.0, 2.0),
        rotation: Quaternion::identity(),
        timestamp: state.t,
        parent: "bike_frame".into(),
        child: "bike_com".into(),
    };

}
